系统学 Codex 终端

不要从“背命令”开始。更快的路线是先建立心智模型：Codex 终端 = 交互会话、 非交互执行、安全边界、MCP/插件、Skill 工作流、 Git 验证闭环。命令只是入口，真正的能力来自“给清楚目标 + 选对工作流 + 验证结果”。

这页把常见命令、oh-my-codex 的工作流入口、Skill 调用技巧和练习任务放在一起。 你可以照着 6 周路线走，也可以把它当成日常速查表。

Learning Method

学习顺序：先场景，后命令

把 Codex 当成“会使用工具的结对工程师”，而不是单纯聊天框。每学一个命令，都立刻绑定一个真实场景。

① 建立地图

运行 codex --help、codex exec --help、omx --help，先知道入口有哪些。

② 小任务实战

从“解释一个文件”“修一个小 bug”“生成一页文档”开始，不要一上来做大型自动化。

③ 复盘提示词

记录哪些表达让 Codex 更稳定：目标、边界、验收标准、禁止事项、测试命令。

④ 固化为 Skill

当同类任务重复出现，就用现成 Skill 或自己写 Skill，把经验变成可复用流程。

6-Week Roadmap

一条系统学习路线

每天 30–60 分钟即可。每周都要产出一个能运行、能验证、能复盘的小成果。

终端基础与安全边界

  • 掌握 codex、codex resume、codex completion。
  • 理解工作目录 -C、模型 -m、配置覆盖 -c。
  • 练习用 read-only、workspace-write、danger-full-access 描述风险。

读项目与提需求

  • 让 Codex 解释目录结构、调用链、测试入口。
  • 学习写“任务边界”：哪些文件能改、哪些不能改。
  • 用 $deep-interview 或 $plan 处理不清楚的需求。

小改动闭环

  • 让 Codex 修一个 bug、补一个测试、改一个 UI 文案。
  • 要求先看 diff，再跑最相关测试。
  • 复盘：最终回答必须包含 changed files、验证证据、风险。

Skill 调用与组合

  • 练习 $code-review、$security-review、$web-clone、$html-ppt。
  • 学会显式调用：$plan 请为这个重构给出验收标准。
  • 学会组合顺序：先 $plan，再实施，最后 $code-review。

GitHub 与自动化交付

  • 掌握 codex review --uncommitted、gh-fix-ci、gh-address-comments。
  • 练习提交前检查：git status、git diff、测试命令。
  • 发布静态页面、修 CI、回复 review 评论。

沉淀自己的工作流

  • 用 $skill-creator 或本地 SKILL.md 固化重复流程。
  • 把常用提示词变成模板：需求澄清、代码审查、发布说明。
  • 建立个人“任务后复盘”：什么可自动化、什么必须人工确认。
Command Lab

Codex / OMX 常见命令速查

下面按本机 codex --help、codex exec --help、omx --help 摘要整理。点击按钮可复制。

codex "帮我理解这个项目"

启动交互式会话。适合探索、迭代、需要多轮确认的任务。

交互入口TUI
codex exec "生成 README 摘要"

非交互执行一次性任务。适合脚本化、批处理、CI 辅助或明确的小任务。

非交互alias: e
codex resume --last

继续最近一次会话。适合中断后恢复上下文。

恢复会话
codex fork --last

从最近会话分叉一条新思路。适合比较两种方案而不污染原会话。

分叉探索
codex review --uncommitted

审查 staged、unstaged、untracked 改动。适合提交前找逻辑问题。

review提交前
codex review --base main

按指定 base 分支审查差异。适合 PR 前自查。

reviewdiff
codex mcp list

查看外部 MCP server。适合排查工具、数据库、浏览器、GitHub 等连接能力。

MCP工具层
codex completion zsh

生成 shell 自动补全脚本；也支持 bash、fish、powershell、elvish。

补全
codex -C path/to/repo

指定工作根目录。避免 Codex 在错误目录读写文件。

工作目录安全
codex --sandbox workspace-write

设置沙盒策略。学习顺序建议：先 read-only，再 workspace-write，最后才理解 danger-full-access。

sandbox安全边界
codex -a on-request

配置审批策略。学习时推荐知道每种含义，不要盲目长期使用最高权限。

approval
codex --full-auto

低摩擦自动执行别名，适合可信仓库的小任务；高风险任务仍要明确边界。

自动执行
codex --search "查官方文档"

启用实时网页搜索。适合依赖最新文档、价格、规则、版本的信息。

web search
codex -i screenshot.png

给初始提示附图。适合 UI 还原、截图诊断、设计审查。

视觉输入
omx setup / omx doctor

安装与诊断 oh-my-codex。遇到 Skill、MCP、AGENTS 配置问题先 doctor。

OMX诊断
omx explore --prompt "找入口文件"

只读探索入口。适合快速找文件、符号、调用关系。

只读探索
omx sparkshell "rg TODO"

运行嘈杂的只读 shell 查询，适合 repo-wide 搜索、日志摘要。

只读 shell

查看 active modes 和执行轨迹。适合长任务复盘。

状态复盘

查看或设置推理强度。复杂架构和诊断用高一些，简单改动不必浪费。

Skill Invocation

Skill 调用技巧

Skill 的关键不是“会喊名字”，而是知道什么时候该把普通任务升级成结构化工作流。

显式调用模板

当你已经知道要哪种工作流，用 $name 开头最稳。

$plan 请帮我为这个功能制定实现计划。 背景：…… 边界：只能改 src/components 和 tests。 验收：页面可访问、测试通过、最终说明风险。

关键词自动路由

有些词会触发工作流，例如 code review、security review、web-clone、deep interview。想避免误触发，就说清楚“只解释，不执行”。

请做 code review： - 只关注这次 diff - 按严重程度排序 - 不要改代码，先给结论和证据
$plan规划

需求清楚但实现路径还需要拆解时使用。

  • 适合：功能设计、重构路线、测试策略。
  • 不适合：马上能完成的小改动。
$deep-interview澄清

你自己也没想清楚目标、边界、取舍时使用。

  • 适合：产品需求、架构方向、模糊目标。
  • 关键词：不要假设、采访我、gather requirements。
$ralplan共识计划

需求基本明确，但要更严谨地权衡风险时使用。

  • 适合：中大型改动前的设计评审。
  • 产出：PRD、测试规格或结构化计划。
$code-review审查

提交前或合并前找 bug、可维护性问题和隐藏风险。

  • 加一句：只看当前 diff，按严重程度排序。
  • 让它给证据，不要只给风格建议。
$security-review安全

涉及认证、授权、密钥、文件上传、网络边界时使用。

  • 要求列出攻击面、利用条件、修复建议。
  • 不要把普通 lint 问题混进安全结论。
$gh-fix-ciCI

GitHub Actions 失败时用它定位日志和失败检查。

  • 适合：PR 检查失败、测试在 CI 才挂。
  • 先诊断，再批准改动范围。
$web-clone网页

要复刻一个网页、做视觉还原、验证截图时使用。

  • 最好给 URL、目标尺寸、验收截图。
  • 视觉任务要持续截图对比。
$html-ppt演示

要做 PPT、slides、分享稿、Keynote 风格页面时使用。

  • 给受众、时长、页数、风格关键词。
  • 适合生成可键盘导航的 HTML deck。
$skill管理

列出、安装、搜索或管理本地技能。

  • 重复流程多了，就考虑创建自己的 Skill。
  • 把“经验”变成可复用的 SKILL.md。
Prompt Patterns

最稳定的提问结构

给 Codex 的任务越像工程票据，它越容易产出可验证结果。

结构 你要写什么 示例
目标 最终用户能看到或做到什么。 “新增一个 Codex 学习网页并发布到 GitHub Pages。”
边界 哪些文件能改、哪些不能碰、是否允许新增依赖。 “只做静态 HTML/CSS/JS，不加依赖，不覆盖旧页面。”
验收 怎样算完成，跑什么命令证明。 “页面能本地打开，链接可访问，git diff 清晰。”
风险 必须提前提醒的人类决策点。 “不要删除旧站点；如果要推 main，请说明 commit。”

When To Escalate

什么时候不要硬上自动化

真正熟练的人不是永远开最大权限，而是知道什么时候收窄范围。

高风险命令

涉及删除、重置、迁移生产数据、覆盖发布时，明确让 Codex 先解释计划和回滚方式。

需求分叉

当“做 A 还是做 B”会影响很多文件时，先用 $deep-interview 或 $ralplan。

新 API / 新 SDK

要求先查官方文档，再实现。尤其是 OpenAI、云服务、支付、安全相关能力。

学习建议：每次让 Codex 做事，都按“目标 → 边界 → 验收 → 验证证据”来写。命令会忘，闭环不会。
