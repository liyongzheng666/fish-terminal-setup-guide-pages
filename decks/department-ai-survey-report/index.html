KEY GAPS

亟待增强和优化的地方：主矛盾在可信度，不在入口

不理解业务上下文 · 10 / 14（71%）

这是当前最突出的顾虑。说明如果 AI 无法结合代码库、模块职责和业务背景，就很难真正进入研发主流程。

需要补足：代码上下文、模块说明、内部知识

结果不准确 · 9 / 14（64%）

团队担心答案看起来“像对的”，但并不可靠，因此更适合先在可校验、可回溯的场景中推进。

建议优先：代码理解、注释生成、骨架生成等可验证任务

容易误导排查方向 · 8 / 14（57%）

在 Bug 排查和问题定位场景里，一旦误导方向，时间成本可能比不用 AI 更高。

需要机制：人审、交叉验证、结果留痕

次级顾虑

  • 学习成本高：4 / 14（29%）
  • 生成代码质量不稳定：4 / 14（29%）
  • 责任边界不清、接入使用麻烦：各 1 / 14

管理层可抓的重点

  • 先补 上下文，不要只给一个通用聊天入口。
  • 先做 可信场景，不要一开始押重度自动化排障。
  • 给出 使用规范，让 AI 输出能被复核、能被追踪。
当前真正制约推广的不是“有没有工具”，而是“能不能信”

这一页建议明确告诉经理层，当前最大的阻碍不是入口问题。从问卷看，大家并不排斥使用，也不缺工具入口，真正的问题是输出是否可靠。

三个高频顾虑连起来其实是一件事：不理解上下文，导致结果不够准，进一步就可能误导排查方向。所以主矛盾本质上是可信度问题。

这也决定了推进策略。我们应该优先做那些更容易校验、更容易积累上下文的场景，而不是一开始就让 AI 在高风险排障场景里“自主发挥”。

STAGE JUDGMENT

当前阶段更适合先做知识库地基，而不是先做 Agent

为什么当前不宜把重点放在 Agent 上

  • 目前缺少专门开发 Agent 的人力和持续维护能力，直接上复杂编排风险较高。
  • 如果没有模块知识、错误经验和历史证据，Agent 只能做表面解释，很难真正看懂老代码。
  • 在老旧代码场景里，决定效果的往往不是“壳子”，而是背后的上下文质量。

当前阶段更现实的目标

  • 先把旧代码相关知识整理成 可检索、可更新、可被模型消费 的内部知识库。
  • 先支持“问得到、查得到、解释得出来”，再考虑复杂的自动编排和多 Agent。
  • 先把最痛的 1 到 2 个老模块梳理清楚，比一开始铺全局平台更务实。

结合现有条件，最小可行路径已经具备

  • 环境：Win10 + 纯内网，天然适合走内网知识库方案。
  • 模型：已有 OpenAI 格式自部署模型，可直接作为问答和总结能力底座。
  • 入口：前期不必自研复杂 Agent 平台，先用简单终端 / IDE 查询入口即可。

当前阶段真正要先做出来的能力

  • 模块职责、关键流程、常见故障、历史改动这些信息能被问到、查到、引用到。
  • 现有模型基于知识库先完成解释、摘要、问答、候选推断等轻量能力。
  • 等知识库足够可用后，再决定是否值得往 Agent 和自动化方向继续投入。
结论：对当前阶段来说，最优解不是先做一个复杂 Agent，而是先把 旧代码知识库 做成可用地基，再让现有模型基于这层地基发挥作用。
这一页回答“为什么当前阶段要先做知识库地基”

这一页的核心不是否定 Agent，而是说明在当前人力和阶段下，先做 Agent 并不是最优投入方向。

如果领导追问为什么，就直接讲两点：第一，没有足够知识底座时，Agent 只是外壳；第二，复杂编排需要持续投入，而你们当前更缺的是老代码上下文本身。

所以这一页要得出的结论是：先把知识库地基做出来，用现有模型把解释、摘要、问答先跑通，后面再评估是否值得继续做 Agent。

KNOWLEDGE BASE

现阶段更值得投入的事：把旧代码知识库做成基础可用

必须先准备的 4 类资产

  • 模块地图：目录范围、职责、入口、关键结构、上下游依赖。
  • 代码层级结构：接口层、流程层、算法层、适配层、配置层、日志层。
  • 常见错误点清单：错误现象、触发条件、先看哪里、常见误判、修复注意事项。
  • 历史证据：git 变更、相似实现、旧版行为、配置差异、典型日志。

为什么这些东西必须要有

  • 老代码的设计意图往往已经不在“人”那里，而散落在代码结构和历史痕迹里。
  • AI 只有拿到这些结构化上下文，才有可能从“解释代码”升级到“推断意图”。
  • 这不是写文档，而是在建设 AI 可消费的工程上下文。

建议的数据结构长什么样

  • 模块卡：模块职责、关键入口、核心流程、关键配置、主要风险点。
  • 错误卡：现象、日志、错误码、优先排查文件、常见根因、误判点。
  • 流程卡：某功能从入口到输出的关键路径与关键节点。
  • 配置卡：宏开关、环境差异、行为分叉点。

建议怎么开始，并为后续演进预留空间

  • 先挑 1-2 个老模块，围绕代码理解和常见故障做试点。
  • 优先补 最常接手、最常排障、最晦涩 的区域。
  • 第一阶段先用 简单终端 / IDE 查询入口 + 现有模型 完成问答、解释、摘要等轻量能力。
  • 每次读码、排障、重构结束后，把结论反向沉淀为模块卡和错误卡。
  • 等知识库足够稳定后，再评估是否值得演进到 终端 + Agent 的代码理解工作台。

补充建议

对你们来说，当前最务实的目标不是“大而全”，而是先梳理出一个 基本可用的旧代码知识库；这层地基稳定后，再逐步演进到 终端 + 模型 + Agent 的代码理解工作台，会更稳、更可落地。

这一页回答“这个知识库具体该准备什么、怎么开始做”

这一页就是把“地基”讲实。既然当前阶段决定先不做复杂 Agent，那这一页就要明确告诉领导，知识库不是泛泛写文档，而是要沉淀哪些具体资产。

建议直接说 4 类：模块地图、代码层级结构、常见错误点清单、历史证据。再补一句，这些是为了让现有模型能真正围绕老代码做解释和推断。

同时把后续方向也留住。可以明确讲，终端和 Agent 不是不要做，而是放在下一阶段：等知识库基础可用后，再往代码考古 Agent、调用链 Agent、故障经验 Agent 这些能力演进。
