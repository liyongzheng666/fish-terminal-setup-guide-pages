01

问卷分析与汇报定位

这一阶段的目标，是把一份调查问卷结果整理成面向部门经理的汇报框架，并严格围绕问卷原始内容做现状、亮点和待优化点分析。

CSV → 大纲 → 逐页文案
U
User起始需求

请根据部门 AI 使用情况调查问卷结果，整理一份可以向部门经理汇报的 PPT 大纲，要求内容严格按照调查问卷来，分析当前现状、做得好的方向以及需要亟待增强和优化的地方。

A
Assistant分析路径

先读取问卷原始字段和答卷内容，核对题目结构、样本分布和多选题热点，再按“使用现状、优势方向、优先建设点”三条线重新归纳成汇报框架。

·
Milestone数据结构化后得到的核心结论

样本共 14 份，主体为内核 / C++ 研发；需求最集中在“阅读和解释陌生代码”“代码重构建议”“根据需求生成代码骨架”；主要顾虑集中在“上下文不足、结果不准、误导排查”。

A
Assistant首版输出

给出 9 页经理汇报大纲，包括：样本画像、当前现状、工作重心、需求全景、优先落地场景、目前做得好的方向、亟待增强和优化的地方、汇总结论与建议。

U
User迭代要求

我看到大家还都给出了一些日常实践，请加到合适的 PPT 分页中，搞成逐页可汇报文案版本。

A
Assistant深化输出

把开放题里的真实实践并进来，扩展成逐页可汇报文案：每页有标题、结论句、图表建议和口播逻辑，保证既能回到问卷，又能直接面对经理层汇报。

阶段产物

完成从问卷数据到 9 页汇报框架的转译，并把“日常实践”从开放题中抽取出来，嵌入到逐页可汇报文案中。

02

HTML Deck 搭建

在确定内容框架后，把文案变成一份可以在浏览器里直接演示的 HTML 汇报稿，同时保留 presenter notes 和后续迭代空间。

文案 → 9 页 Deck
U
User制作要求

希望把逐页文案做成 HTML PPT。

A
Assistant制作策略

选择正式汇报感更强的 `weekly-report` 模板作为底板，再把问卷结论、逐页文案和 speaker notes 重新排成 9 页 HTML deck。

·
Milestone制作动作

新建 `decks/department-ai-survey-report/` 目录，落地 `index.html`、`style.css`，按页写入统计、开放题案例、讲稿备注和页码。

A
Assistant交付方式

完成后直接在本地打开 Deck，支持方向键翻页、`N` 查看备注、`S` 打开 presenter 视图，方便快速进入“边看边改”的状态。

U
User预览需求

请在我本地打开，让我看一下效果。

A
Assistant进入迭代

本地打开 Deck，开始围绕页面切换、封面排版、整体风格和标题口径做多轮迭代。

阶段产物

形成第一版 HTML 汇报稿，并建立“本地直接预览 + 当场微调”的工作流。

查看最终汇报稿
03

视觉与版式迭代

这一阶段主要围绕“领导观感”和“讲述节奏”做细节打磨，包括过渡动画、封面重排、整体块感减轻和标题口径统一。

观感优化
U
User视觉反馈

我想让切换页面时的风格更多样一些。

A
Assistant动画策略

为不同页面配置不同的切页气质和入场动效，比如 `cover / glide / reveal / cards / spotlight / finale`，让汇报过程更有节奏感，但保持正式克制。

U
User封面反馈

第一页中的框框有重叠，让人看起来很难受；另外统计口径和样本主体也有重合。

A
Assistant版式修复

把封面从“多个重卡片”改成更轻的无边框信息条，去掉重复统计信息和时间，让第一页更像真正的封面，只承担定位和开场作用。

U
User风格方向

后面那几页也改一下吧；字数不需要压，主要用于交流会，领导愿意看细节。另外标题口径统一一下。

A
Assistant统一风格

统一收轻后续页面的区块样式，把厚重卡片感降下来；同时把页标题改成同一套交流会口径，让全篇更稳、更专业。

阶段产物

Deck 从“能看”进一步变成“可正式展示”：封面更干净，后续页面更轻，标题口径统一，整体更适合交流会使用。

04

旧代码理解方案深化

在基本汇报稿成型后，重点新增了“如何用 AI 解决老旧代码理解问题”的章节，并结合团队真实阶段把路径逐步收敛到“先做知识库地基”。

方案深化
U
User新增主题

为了实现老旧代码的理解，我们可以做些什么？当前环境是 Win10、纯内网，但已有 OpenAI 格式自部署模型。

A
Assistant初始方案

提出“代码考古”思路：目标不是一次性还原作者想法，而是让 AI 基于代码结构、调用关系、日志、配置和历史痕迹做事实解释、设计意图推断与不确定点标注。

U
User技术追问

我们更想先搞定工具，需要一个读代码能力强一点的终端软件。另外是不是需要准备模块常见错误点、代码层级结构？希望展开，而且这些东西都要真正能落地。

A
Assistant方法框架

明确指出：终端只是入口，真正决定效果的是模型层、代码情报层、证据检索层和知识沉淀层；并展开说明模块地图、代码层级结构、常见错误点清单、历史证据等结构化资产的重要性。

U
User阶段约束

我们并没有开发 Agent 的能力和人员，我想的是先把第十一页做好，搞一个好的地基，梳理出一个基本可用的知识库更符合我们当下的阶段。

A
Assistant路径收敛

把后两页重构成：第 10 页先回答“为什么当前阶段先不做 Agent，而要先做知识库地基”；第 11 页再展开“这个知识库具体该准备什么、怎么开始做”，同时把 `终端 + Agent` 作为后续演进方向保留。

阶段产物

新增并反复打磨了“旧代码理解”章节，最终形成了“当前先做知识库地基、未来再考虑终端 + Agent 工作台”的可落地路线。

05

发布到 GitHub

为了实现“随时随设备查看”，在内容确认后将 deck 改造成仓库内自包含静态目录，并推送到 GitHub Pages。

本地 → GitHub Pages
U
User发布需求

想把这个内容先搞到 GitHub 上，可以随时随设备查看，请帮我完成。

A
Assistant先做检查

先检查仓库 remote、当前分支、GitHub 身份以及 Pages 发布条件，然后确认 deck 的样式与运行时资源能否脱离本机环境独立工作。

·
Milestone发布前修正

发现 deck 目录里的 `assets` 是指向本机 `.codex` 路径的符号链接，如果直接推上 GitHub，外部设备将无法加载样式和交互。于是先将运行所需素材复制进仓库，确保 GitHub Pages 可独立运行。

A
Assistant发布动作

将 `decks/department-ai-survey-report/` 整体提交到仓库，按 Lore 协议写提交信息，然后推送到 `main`。

·
Milestone发布结果

GitHub Pages 最新构建状态为 `built`，生成可直接访问的在线链接，方便在手机、平板和其他电脑上随时查看。

阶段产物

HTML 汇报稿已作为自包含静态目录发布到 GitHub Pages，可跨设备访问。
